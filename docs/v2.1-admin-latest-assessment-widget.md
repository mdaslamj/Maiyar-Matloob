# Version 2.1 — Admin Live Widget: Latest Assessment

Phase 3 connects **one** additional dashboard widget to Firestore.

## Widget

| Label | Source | Status |
|-------|--------|--------|
| Latest Assessment | Most recent `assessments/` document by `timestamp` | Live when `USE_FIREBASE: true` |

Total Participants and Completed Assessments remain live from earlier phases.
All other dashboard cards remain sample data.

## Feature flag

| Flag | Behavior |
|------|----------|
| `USE_FIREBASE: false` | Sample `summary.latestAssessmentDate` |
| `USE_FIREBASE: true` | Live Firestore query via `admin-firestore-service.js` |

When live mode is enabled, this widget never shows sample data.

## Architecture

```
admin-dashboard.js
        ↓
admin-dashboard-data.js
        ↓
admin-firestore-service.js → loadLatestAssessmentWidgetData()
        ↓
Firestore: assessments/ (limit 1)
```

UI components never query Firestore directly.

## Firestore query

```javascript
query(
  collection(db, "assessments"),
  orderBy("timestamp", "desc"),
  limit(1)
)
```

Only **one document** is read. The full collection is never downloaded.

Timestamp resolution order:

1. `timestamp` (Firestore server timestamp)
2. `capturedAt` (ISO fallback on older records)

## Display

| Field | Source |
|-------|--------|
| Main value | Assessment date (`formatDate`) |
| Meta line | Relative label + time (`Today, 14:32` / `Yesterday, 09:15` / `14:32`) |

UI card style unchanged (`renderMetricCard`).

## Loading behavior

1. Dashboard renders `Loading...` for Latest Assessment
2. `loadDashboardLiveWidgets()` fetches all live widgets in parallel
3. Card updates with date + relative/time meta on success

## Empty state

| Value | Meta |
|-------|------|
| `—` | `No assessments submitted yet.` |

## Error handling

| Status | Value | Meta |
|--------|-------|------|
| `permission-denied` | `—` | `Permission denied.` |
| `unavailable` | `—` | `Firestore unavailable.` |
| `error` | `—` | `Unable to load latest assessment.` |

Errors are caught in the service layer. The dashboard never crashes.

## Security requirements

Admin read access on `assessments/` was added in Phase 2:

```
allow read: if (own uid) || isGoogleAdmin();
```

Participant rules unchanged. No rule changes required for Phase 3.

## Firestore indexes

Query:

```
orderBy("timestamp", "desc")
limit(1)
```

This is a **single-field** ordering query on a collection. Firestore creates automatic single-field indexes by default.

**No additional composite index is required** for Phase 3.

Existing indexes in `firestore.indexes.json` (`uid + timestamp`, `overallLevel + timestamp`) remain for future filtered queries only.

If Firestore returns a `failed-precondition` index error in a specific project configuration, add:

```json
{
  "collectionGroup": "assessments",
  "queryScope": "COLLECTION",
  "fields": [
    { "fieldPath": "timestamp", "order": "DESCENDING" }
  ]
}
```

## Performance notes

- Reads exactly **one document**
- Runs in parallel with other live widgets via `Promise.all`
- No aggregation scan and no full collection download
