# Version 2.1 — Admin Live Widget: Average Implementation

Phase 4 connects **one** additional dashboard widget to Firestore using a **locked business definition**.

## Widget

| Label | Source | Status |
|-------|--------|--------|
| Average Implementation | Latest assessment score per participant | Live when `USE_FIREBASE: true` |

Earlier live widgets (Total Participants, Completed Assessments, Latest Assessment) are unchanged.
All other dashboard cards remain sample data.

---

## Business definition (locked)

**Average Implementation** = the community's **current implementation level**.

Each participant (`uid`) contributes **exactly one value**: the overall implementation score from their **most recent** completed assessment.

Historical assessments for the same participant are **ignored**.

### Formula

```text
Average Implementation
=
Sum(latest implementation score of each participant)
÷
Number of participants with at least one assessment
```

### Example

| Participant | Latest score |
|-------------|--------------|
| A | 82% |
| B | 64% |
| C | 91% |
| D | 58% |

**Average** = (82 + 64 + 91 + 58) ÷ 4 = **73.8%**

### Why not average all assessments?

A participant who completes many assessments would otherwise dominate the community average. Latest-per-participant gives **equal weight** to each community member.

```
admin-dashboard.js
        ↓
admin-dashboard-data.js
        ↓
admin-firestore-service.js → loadAverageImplementationWidgetData()
        ↓
community-implementation-average.js (pure business logic)
        ↓
Firestore: assessments/ (paginated, ordered desc)
```

UI components never query Firestore directly.

## Business logic module

`src/firebase/community-implementation-average.js`

| Function | Role |
|----------|------|
| `extractImplementationScore()` | Reads `implementationScore` or `overallPercentage` |
| `mergeLatestImplementationScores()` | Keeps first score per uid from desc-ordered batch |
| `calculateCommunityAverageImplementation()` | Mean of latest scores |

---

## Firestore query strategy

Paginate assessments in **descending timestamp order**:

```javascript
query(
  collection(db, "assessments"),
  orderBy("timestamp", "desc"),
  limit(200),
  startAfter(lastDoc) // next page
)
```

Processing rule:

1. Walk pages newest → oldest
2. For each document, if `uid` not yet seen, record score (this is that participant's latest)
3. Skip older documents for uids already recorded
4. Compute mean of recorded latest scores

Score field: `implementationScore` (fallback `overallPercentage`).

## Performance considerations

### Current approach (Phase 4)

- Paginated reads of `assessments/` ordered by timestamp desc
- Stops after all pages processed
- Each assessment document is read **once** during calculation
- Acceptable for early community scale

### Future optimization options (not implemented)

1. **Precomputed aggregate** at `admin/dashboard/metrics.averageImplementation`, updated on each assessment sync
2. **Per-participant latest snapshot** document updated on submit (avoid full scan)
3. **Cloud Function** to maintain community metrics asynchronously

Do **not** introduce Cloud Functions in Phase 4.
