## 1. Executive Summary

### 1.1 Purpose

This document is the long-term architectural reference for Maiyar-e-Matloob backend integration. It defines how the application persists, retrieves, and secures data across multiple backend technologies without changing business logic, UI, scoring, or reporting.

### 1.2 Problem Statement

The current codebase couples cloud integration directly to Firebase (Firestore sync, Firebase Auth, provider-specific admin reads). This creates vendor lock-in, prevents environment portability, and mixes persistence concerns with domain logic.

### 1.3 Solution

Introduce a **provider-independent backend architecture** composed of:

1. An explicit **Domain Layer** owning all business rules
2. A **Backend Adapter Layer** split into independent **Identity** and **Storage** adapters
3. A **Canonical Data Model** as the single source of truth for all providers
4. Pluggable **Backend Providers** implementing the adapter contracts

### 1.4 Approved Constraints

The following remain unchanged by this architecture:

| Layer | Status |
|-------|--------|
| Presentation (welcome, questionnaire UI, admin UI appearance) | Frozen |
| Application orchestration (`application.js`, admin bootstrap) | Wiring targets change; behavior does not |
| Assessment Engine, Scoring Engine, Report Generation | Frozen |
| Trend Engine, Recommendation Engine, Insights | Frozen |
| Business rules (e.g. Average Implementation = mean of latest score per participant) | Locked |

### 1.5 Provider Strategy

| Provider | Role |
|----------|------|
| **Supabase** | First backend implementation (PostgreSQL + Auth + RLS) |
| **Firebase** | Supported legacy provider; preserved behind adapter |
| **REST API** | Future provider via HTTP contract |
| **Self-hosted** | Future provider via custom server or direct database access |
| **Future providers** | Registered without architectural redesign |

Supabase is an **implementation**, not the architectural center. The adapter contracts and canonical model are the center.

### 1.6 Persistence Philosophy

> **The canonical persistence mechanism is provided by the active backend provider.** Client-side caching, offline storage, or browser persistence are implementation details and are not part of the architectural contract.

The application may use IndexedDB, browser cache, memory cache, offline sync, or no local cache — these choices do not alter the architecture. The backend provider is the authoritative store when sync is enabled.

## 2. Architecture Principles

These principles govern all backend-related decisions for the life of the project.

### 2.1 Separation of Concerns

Each layer has a single responsibility. Presentation renders. Application orchestrates. Domain computes. Adapters translate. Providers persist.

### 2.2 Domain-Driven Design

Business rules live exclusively in the Domain Layer. No domain module may import a backend provider, adapter implementation, or persistence SDK.

### 2.3 Provider Independence

Application and Domain layers depend on adapter **interfaces**, not on Supabase, Firebase, or any specific technology.

### 2.4 Single Source of Truth

The **Canonical Data Model** defines entities, field names, types, and relationships. All providers map to and from this model. Provider-specific terminology never appears in domain or application contracts.

### 2.5 Replaceable Infrastructure

Identity and Storage adapters are independently replaceable. A deployment may use Supabase for storage and a future OAuth provider for identity, or swap storage without changing identity configuration.

### 2.6 Backward Compatibility

Existing participant reports, scoring outputs, and admin widget business rules remain valid. Provider migration transforms data at the canonical schema level; business semantics do not change.

### 2.7 Minimal Vendor Lock-in

Provider SDKs are confined to `src/backend/providers/{name}/`. No provider import crosses the adapter boundary.

### 2.8 Long-term Maintainability

Prefer explicit contracts, canonical naming, and documented mapping tables over implicit conventions tied to one vendor.

### 2.9 Simplicity over Complexity

One facade (`backend-service.js`) for application consumption. Two adapter interfaces (Identity, Storage). No unnecessary abstraction layers beyond these.

### 2.10 Security by Design

Authorization policies are defined at the canonical level first. Providers implement equivalent enforcement. Business authorization never depends on a specific provider's syntax.

### 2.11 Scalability without Architectural Rewrite

Adding providers, environments, organizational roles, or new persistence operations extends the adapter contract. It does not require redesigning Domain, Application, or Presentation layers.

## 3. Layered Architecture

### 3.1 Layer Stack

```text
Presentation Layer
        │
Application Layer
        │
────────────────────────────
        │
Domain Layer
────────────────────────────
        │
Backend Adapter Layer
        │
 ┌───────────────┬────────────────┐
 │               │
Identity Adapter Storage Adapter
 │               │
 └───────────────┴────────────────┘
        │
Backend Provider
        │
├── Supabase
├── Firebase
├── REST API
├── Self Hosted
└── Future Providers
```

### 3.2 Layer Responsibilities

| Layer | Owns | Must not own |
|-------|------|--------------|
| **Presentation** | UI rendering, user input, accessibility | Business rules, persistence, auth SDK calls |
| **Application** | Session orchestration, engine coordination, sync triggers | Provider-specific queries, scoring formulas |
| **Domain** | Business rules, calculations, canonical entity construction | HTTP calls, database drivers, OAuth popups |
| **Backend Adapter** | Interface contracts, provider registry, canonical mapping orchestration | Business rule computation |
| **Backend Provider** | Native SDK/API calls, provider-specific security enforcement | UI logic, scoring, report generation |

### 3.3 Dependency Rules

```text
Presentation  →  Application  →  Domain
Application   →  Backend Adapter (via backend-service facade)
Backend Adapter  →  Provider implementations only
Domain  ↛  Backend Adapter (domain never calls providers)
Domain  ↛  Provider SDKs
Providers  ↛  Domain, Application, Presentation
```

### 3.4 Current State vs Target State

**Today (pre-v2.2):** Application layer imports Firebase modules directly. Community average logic lives under `src/firebase/`. Admin reads query Firestore directly via `admin-firestore-service.js`.

**Target (v2.2):** Application imports `backend-service.js` only. Domain logic consolidated under explicit domain modules. Firebase code moves to `providers/firebase/` behind Storage and Identity adapters.

## 4. Domain Layer

### 4.1 Purpose

The Domain Layer owns Maiyar-e-Matloob's business rules and analytical logic. It transforms inputs (questionnaire answers, historical snapshots, community records) into domain outputs (assessments, reports, trends, aggregates, recommendations).

The backend exists **only to persist and retrieve domain data**. It does not compute business outcomes.

### 4.2 Ownership Boundaries

| Domain Service | Responsibility | Current Location |
|----------------|----------------|------------------|
| **Assessment Engine** | Question scoring, raw totals, category percentages | `src/assessment/assessment.js` |
| **Scoring Engine** | Per-question score, reverse scoring, max score resolution | `src/assessment/assessment.js` |
| **Report Generation** | Strengths, weaknesses, insights, recommendations | `src/report/report.js` |
| **Trend Engine** | Historical comparison, growth detection | `src/trend/` |
| **Recommendation Engine** | Action plans, reassessment guidance | `src/action/`, `src/reassessment/` |
| **Feedback Engine** | Urdu guidance generation | `src/feedback/` |
| **Community Implementation Logic** | Average implementation calculation, latest-per-participant merge | `src/firebase/community-implementation-average.js` → moves to domain |
| **Participant Lifecycle** | Profile validation, assessment count tracking | `src/core/application.js` (orchestration) + participant storage contracts |
| **Analytics Calculations** | Community bucket mapping, section aggregates | Domain transforms; storage adapter persists |
| **History Intelligence** | Snapshot building, immutable history reads | `src/history/` |
| **Insights Selection** | Personal insights, dashboard view models | `src/insights/`, `src/growth/`, `src/dashboard/` |

### 4.3 Domain Module Organization (Target)

```text
src/domain/                          # Explicit domain layer (target structure)
├── assessment/                      # Re-exports or wraps src/assessment (frozen logic)
├── report/                          # Re-exports or wraps src/report
├── trend/                           # Re-exports or wraps src/trend
├── community/
│   ├── implementation-average.js    # Average Implementation business rules
│   └── aggregate-transforms.js      # Bucket mapping, count shapes
├── participant/
│   └── participant-lifecycle.js     # Domain rules for participant state
└── analytics/
    └── dashboard-metrics.js         # Domain calculations for admin widgets
```

**Migration note:** Existing engine modules under `src/assessment/`, `src/report/`, etc. remain frozen. v2.2 introduces explicit domain boundaries and moves provider-adjacent logic (e.g. community average) out of `src/firebase/` into `src/domain/`. Full physical consolidation of all engines into `src/domain/` may occur incrementally; the architectural rule is enforced by import boundaries, not folder renames alone.

### 4.4 Domain Rules (Locked Examples)

**Overall implementation percentage:**

```text
overallPercentage = (sum of question scores / sum of max scores) × 100
Rounded to one decimal place.
implementationScore = overallPercentage (identical; not a separate metric).
```

**Average Implementation (community):**

```text
For each participant: take the implementation score from their most recent assessment.
Average Implementation = arithmetic mean of those latest scores.
One participant = one contribution. Historical assessments are ignored.
```

**Community answer aggregates:**

```text
Each submission increments anonymous bucket counts (always, often, sometimes, never).
No individual answers stored in community aggregate entities.
```

### 4.5 Domain Input/Output Contracts

Domain modules accept and return **canonical entities** (Section 6) or pure value objects. They never receive Firestore documents, SQL rows, or REST response shapes directly — adapters map to canonical form before domain consumption.

Example flow for Average Implementation widget:

```text
Storage Adapter  →  returns canonical Assessment[] (raw rows)
Domain Layer     →  mergeLatestScores() + calculateAverage()
Application Layer →  formats widget result for admin UI
```

## 5. Backend Adapter Layer

### 5.1 Overview

The Backend Adapter Layer is split into **two independent adapters** plus a **facade** that application code consumes.

```text
backend-service.js          ← Application imports this only
        │
backend-adapter.js          ← Provider registry + lifecycle
        │
   ┌────┴────┐
   │         │
identity-   storage-
adapter     adapter
   │         │
   └────┬────┘
        │
   Provider implementations
```

Identity and Storage are replaceable independently. A deployment may configure different providers for each in future versions; v2.2 implementations bundle both under a single named provider (e.g. `supabase`, `firebase`).

### 5.2 Identity Adapter

**Purpose:** Establish and validate identities for participants and administrators.

**Responsibilities:**

| Operation | Description |
|-----------|-------------|
| `initialize()` | Init auth SDK/client |
| `ensureParticipantIdentity()` | Silent anonymous identity for participant sync (no UI) |
| `getParticipantIdentityId()` | Return current participant identity identifier |
| `signInAdminWithOAuth(provider)` | Administrator OAuth (Google initially) |
| `signOut()` | End current session |
| `requireAdminSession()` | Validate admin session and authorization |
| `watchSession(callback)` | Session state change listener |
| `isAuthorizedAdmin(email)` | Client-side allowlist check |
| `getSupportedAuthProviders()` | List available auth methods |

**Future extension points:**

- Organization SSO (SAML, OIDC)
- Multi-factor authentication for administrators
- Role-based identity claims (organization admin, reviewer)
- Token refresh and session expiry policies

**Participant UX constraint:** Participant identity establishment must remain frictionless — no login screens for assessment completion unless explicitly required by a future organizational policy.

### 5.3 Storage Adapter

**Purpose:** Persist and retrieve canonical entities. No business rule computation.

**Responsibilities:**

| Operation | Canonical Entity | Description |
|-----------|------------------|-------------|
| `saveParticipant(participant)` | Participant | Upsert participant profile |
| `saveAssessment(assessment)` | Assessment | Persist completed assessment |
| `saveReport(report)` | Report (embedded in Assessment) | Full report snapshot |
| `incrementCommunityAggregates(record)` | CommunityAnswerRecord | Atomic aggregate increments |
| `recordClientVersion(meta)` | SystemVersion | Client telemetry |
| `countParticipants()` | — | Admin widget: total participants |
| `countAssessments()` | — | Admin widget: completed assessments |
| `loadLatestAssessment()` | Assessment | Admin widget: most recent by timestamp |
| `loadAssessmentsForAverage(options)` | Assessment[] | Paginated, ordered by timestamp desc |
| `loadCommunityQuestionAggregates()` | CommunityQuestionAnswer[] | Future insights |
| `loadCommunitySectionAggregates()` | CommunitySectionAggregate[] | Future insights |
| `loadParticipantHistory(participantId)` | Assessment[] | Future admin/history features |
| `loadAdminDashboardMetrics()` | AdminDashboardMetrics | Future metrics placeholder |

**Write behavior:** All writes accept canonical payloads from `backend-schema.js`. Provider translates to native format.

**Read behavior:** All reads return canonical entities. Domain layer performs calculations on returned data.

**Widget results:** Storage adapter returns raw canonical data. Application/Domain layers produce normalized widget results:

```javascript
{
  status: "loading" | "success" | "empty" | "error" | "unavailable" | "permission-denied",
  value,
  message,
  lastUpdated
}
```

### 5.4 Backend Service Facade

`backend-service.js` is the **only module** imported by:

- `src/core/application.js` (participant sync)
- `src/admin/admin-dashboard-data.js` (admin reads)
- `src/admin/admin-auth-gate.js` (admin auth)

The facade delegates to Identity and Storage adapters, handles initialization order (identity before storage writes), and exposes a stable API that survives provider changes.

### 5.5 Provider Registry

```javascript
// Conceptual registry (not implementation code)
PROVIDERS = {
  supabase:  { identity: SupabaseIdentityAdapter,  storage: SupabaseStorageAdapter },
  firebase:  { identity: FirebaseIdentityAdapter,  storage: FirebaseStorageAdapter },
  rest:      { identity: RestIdentityAdapter,      storage: RestStorageAdapter },
  selfHosted:{ identity: SelfHostedIdentityAdapter,  storage: SelfHostedStorageAdapter },
  none:      { identity: NoOpIdentityAdapter,        storage: NoOpStorageAdapter }
}
```

Selection via environment configuration (Section 8).

### 5.6 Provider Implementation Structure

```text
src/backend/
├── backend-service.js
├── backend-adapter.js
├── identity-adapter-interface.js
├── storage-adapter-interface.js
├── backend-schema.js
├── backend-config.js
├── backend-config.local.example.js
└── providers/
    ├── supabase/
    │   ├── supabase-identity.js
    │   └── supabase-storage.js
    ├── firebase/
    │   ├── firebase-identity.js
    │   └── firebase-storage.js
    ├── rest/
    │   ├── rest-identity.js
    │   └── rest-storage.js
    └── self-hosted/
        ├── self-hosted-identity.js
        └── self-hosted-storage.js
```

## 6. Canonical Data Model

### 6.1 Single Source of Truth

The Canonical Data Model is defined in `backend-schema.js`. It uses:

- **camelCase** field names
- **ISO 8601** timestamps
- **Explicit schemaVersion** on all persistable entities
- **No provider terminology** (no `uid` vs `participant_id` in canonical form — always `participantId`)

All providers map to and from this model at the adapter boundary.

### 6.2 Core Entities

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string (uuid) | yes | Identity-linked participant identifier |
| `localParticipantId` | string | yes | Client-generated local identifier |
| `name` | string | yes | Display name |
| `mobile` | string | no | Contact number |
| `createdAt` | ISO 8601 | yes | Profile creation time |
| `lastAssessmentAt` | ISO 8601 | no | Most recent assessment time |
| `assessmentCount` | integer | yes | Completed assessment count |
| `schemaVersion` | integer | yes | Canonical schema version |

#### Assessment

### 6.5 Query Requirements (Provider-Agnostic)

| Pattern | Used By |
|---------|---------|


## 7. Security Architecture

### 7.1 Policy-First Design

Security is defined at the **canonical policy level**. Providers implement equivalent enforcement through their native mechanisms. Business authorization never depends on provider-specific syntax.

### 7.2 Identity Roles

| Role | Description | Current Support |
|------|-------------|-----------------|
| **Anonymous Participant** | Silent identity for assessment submission | Yes |
| **Authenticated Participant** | Named participant with verified identity | Future |
| **Administrator** | Google OAuth with allowlist | Yes |
| **Organization Admin** | Manages organizational participants and reports | Future |
| **Reviewer** | Read-only access to aggregated data | Future |

### 7.3 Canonical Permission Matrix

#### Anonymous Participant

| Resource | Create | Read | Update | Delete |
|----------|--------|------|--------|--------|
| Own Participant profile | yes | own | own | no |
| Own Assessments | yes | own | no | no |
| Community aggregates | increment | no | increment | no |
| Other participants' data | no | no | no | no |
| System config | no | limited | no | no |
| Admin metrics | no | no | no | no |

#### Authenticated Participant (Future)

Same as anonymous participant, plus potential access to verified profile linking and organizational resources per organization policy.

#### Administrator

| Resource | Create | Read | Update | Delete |
|----------|--------|------|--------|--------|
| All participants | no | yes | no | no |
| All assessments | no | yes | no | no |
| Community aggregates | no | yes | no | no |
| System config | yes | yes | yes | no |
| Admin metrics | yes | yes | yes | no |

Administrators cannot modify participant assessment records (immutable submission model).

#### Organization Admin (Future)

Read/write scoped to organizational participant subset. Policy definition deferred to organizational features phase.

### 7.4 Provider Enforcement Mechanisms

| Provider | Identity Enforcement | Storage Enforcement |
|----------|---------------------|---------------------|
| **Supabase** | Supabase Auth (anonymous + OAuth) | Row Level Security (RLS) policies |
| **Firebase** | Firebase Auth | Firestore Security Rules |
| **REST API** | JWT/OAuth middleware | Server-side authorization middleware |
| **Self-hosted** | Custom auth service | Application-level authorization layer |

### 7.5 Administrator Authorization Model

Dual allowlist pattern (client + server):

1. **Client allowlist:** `ADMIN_ALLOWED_EMAILS` in gitignored `backend-config.local.js` — prevents unauthorized OAuth attempts from proceeding in UI
2. **Server allowlist:** `SystemConfig.adminEmails` — enforced by provider security policies

Both must match for production deployments. Server policy is authoritative.

### 7.6 Participant Identity Recommendation

**Recommended (production):** Silent anonymous identity via provider auth SDK. Establishes `participantId` linked to auth identity. Enables secure row ownership without participant login UI.

**Alternative (restricted environments):** Client-generated identifier without server auth. Requires permissive insert policies. Document as non-production only.

### 7.7 Secrets Management

| Secret | Location | Rule |
|--------|----------|------|
| Provider public/anon key | gitignored local config | Allowed in client SPA |
| Service role / admin credentials | Server-side only | Never in static SPA |
| OAuth client secrets | Provider dashboard / server | Never in client bundle |
| Admin allowlist | Client config + server config | Must stay synchronized |

## 8. Environment Configuration

### 8.1 Configuration Model

Backend behavior is controlled by environment configuration, not code branches in UI or domain layers.

```javascript
// Conceptual configuration (feature-flags.js + backend-config.local.js)
BACKEND_PROVIDER: "none" | "supabase" | "firebase" | "rest" | "self-hosted"
USE_BACKEND_SYNC: boolean
ENVIRONMENT: "development" | "staging" | "production"
```

### 8.2 Configuration Sources (Priority Order)

1. `backend-config.local.js` (gitignored; per-developer and per-deployment)
2. `src/shared/feature-flags.js` (defaults)
3. Build-time environment variables (documented; injected at deploy)

### 8.3 Behavior Matrix

| BACKEND_PROVIDER | USE_BACKEND_SYNC | Identity | Storage | Admin Live Data |
|------------------|------------------|----------|---------|-----------------|
| `"none"` | `false` | NoOp | NoOp | Sample data |
| `"supabase"` | `true` | Supabase Auth | Supabase PostgreSQL | Live widgets |
| `"firebase"` | `true` | Firebase Auth | Firestore | Live widgets |
| `"rest"` | `true` | REST OAuth/JWT | REST API | Live widgets (when implemented) |
| `"self-hosted"` | `true` | Custom | Custom | Live widgets (when implemented) |

### 8.4 Legacy Flag Deprecation

| Legacy Flag | Replacement |
|-------------|-------------|
| `USE_FIREBASE` | `USE_BACKEND_SYNC && BACKEND_PROVIDER === "firebase"` |
| `USE_FIRESTORE` | Out of scope for v2.2; local client cache adapter remains an implementation detail |

### 8.5 Admin Live Data Gate

```text
isAdminLiveDataEnabled() = USE_BACKEND_SYNC && BACKEND_PROVIDER !== "none"
```

When false, admin dashboard displays sample data. Widget rendering code is unchanged; data source selection is configuration-driven.

## 9. Deployment and Migration Strategy

### 9.1 Deployment Topology

```text
Static SPA (CDN / GitHub Pages / any static host)
        │
        ▼
Backend Service Facade (client-side)
        │
   ┌────┴────────────┬──────────────┐
   ▼                 ▼              ▼
Supabase         Firebase        REST API
(PostgreSQL)     (Firestore)     (custom server)
```

The same SPA artifact deploys to any static host. Backend provider selection is configuration-only.

### 9.2 Environment Deployment

| Environment | Purpose | Configuration |
|-------------|---------|---------------|
| **development** | Local developer testing | Provider project with relaxed policies; optional NoOp provider |
| **staging** | Pre-production verification | Production-equivalent policies; test data |
| **production** | Live deployment | Full security policies; monitored |

Each environment maintains its own provider project credentials in gitignored config.

### 9.3 Switching Providers

Provider switch requires **configuration change only** — no application, domain, or UI code changes:

```text
1. Export data from current provider → canonical format
2. Import canonical data into new provider
3. Update BACKEND_PROVIDER and backend-config.local.js
4. Deploy SPA with new configuration
5. Run verification checklist (Section 12)
```

Identity and Storage adapters for a named provider are bundled. Future versions may allow independent identity/storage provider configuration.

### 9.4 Data Migration

All provider-to-provider migration operates on the **canonical schema**:

```text
Source Provider (native format)
        ↓
Transform to Canonical Model
        ↓
Validate schemaVersion + required fields
        ↓
Transform to Target Provider (native format)
        ↓
Import into Target Provider
```

| Migration Path | Tooling |
|----------------|---------|
| Firebase → Supabase | Firestore export → canonical transform → SQL import |
| Supabase → Firebase | SQL/JSON export → canonical transform → Firestore import |
| Any → REST | Canonical JSON bulk upload API |
| Any → Self-hosted | Provider-specific ETL against canonical schema |

Migration scripts are separate tooling (out of scope for adapter implementation). Canonical model ensures field compatibility.

### 9.5 Canonical Schema Migration

When `CANONICAL_SCHEMA_VERSION` increments:

1. Document field additions, removals, or semantic changes
2. Provide transform functions: `v(N) → v(N+1)`
3. Apply transforms during provider data migration
4. Domain layer validates supported versions on read

Backward compatibility: domain logic supports reading previous schema versions until explicitly deprecated.

### 9.6 Rollback Strategy

| Scenario | Action |
|----------|--------|
| New provider fails verification | Revert `BACKEND_PROVIDER` config to previous provider |
| Data import incomplete | Restore from pre-migration export; keep previous provider active |
| Auth misconfiguration | Disable `USE_BACKEND_SYNC`; admin falls back to sample data; participant app continues via client cache if configured |
| Critical security issue | Revoke provider keys; deploy NoOp provider config; investigate |

Rollback never requires UI or domain code changes — configuration and data restoration only.

### 9.7 Failure Behavior

Backend sync is **non-blocking**. If the provider is unreachable:

- Participant assessment completion proceeds (domain logic executes regardless)
- Client cache (if configured) retains data locally
- Sync retries may be implemented as an adapter concern
- Admin dashboard shows unavailable state for live widgets, or sample data if sync disabled

## 10. Provider Comparison

All providers implement the same Identity and Storage adapter contracts. Selection is a deployment decision, not an architectural decision.

### 10.1 Supabase

| Aspect | Detail |
|--------|--------|
| **Status** | First implementation |
| **Identity** | Supabase Auth (anonymous + OAuth) |
| **Storage** | PostgreSQL with RLS |
| **Advantages** | Relational queries; SQL migrations; RLS; open-source option; strong static SPA support |
| **Limitations** | Requires PostgreSQL operational knowledge; RLS policy complexity |
| **Suitable for** | Production deployments requiring relational data, SQL reporting, and managed infrastructure |

### 10.2 Firebase

| Aspect | Detail |
|--------|--------|
| **Status** | Supported legacy provider |
| **Identity** | Firebase Auth (anonymous + Google) |
| **Storage** | Cloud Firestore |
| **Advantages** | Real-time listeners; existing project integration; document model matches current code |
| **Limitations** | NoSQL query constraints; vendor-specific rules syntax; composite index management |
| **Suitable for** | Existing Firebase deployments; environments where Firestore is already provisioned |

### 10.3 REST API

| Aspect | Detail |
|--------|--------|
| **Status** | Future provider |
| **Identity** | JWT, OAuth 2.0, or API key (server-defined) |
| **Storage** | Any server-side database |
| **Advantages** | Maximum flexibility; language-agnostic server; custom business middleware |
| **Limitations** | Requires building and operating a custom API server |
| **Suitable for** | Organizations with existing API infrastructure; multi-client ecosystems |

### 10.4 Self-Hosted

| Aspect | Detail |
|--------|--------|
| **Status** | Future provider |
| **Identity** | Custom (LDAP, OIDC, local auth) |
| **Storage** | Direct database or internal service |
| **Advantages** | Full data sovereignty; no third-party dependency |
| **Limitations** | Highest operational burden; security responsibility on operator |
| **Suitable for** | Air-gapped environments; strict data residency requirements |

### 10.5 Selection Guidance

| Requirement | Recommended Provider |
|-------------|---------------------|
| Fastest path to production with relational backend | Supabase |
| Existing Firebase investment | Firebase |
| Custom middleware or enterprise integration | REST API |
| Full data sovereignty | Self-hosted |

Application code is identical across all choices.

## 14. Architectural Constraints and Governance

This section defines the **permanent architectural boundaries** for Maiyar-e-Matloob. These rules must not be violated unless this document is formally revised and re-approved.

This section is the **constitution** for backend architecture. It exists to prevent architectural drift during implementation and all future development.

### 14.1 Governance Scope

| In scope | Out of scope (requires formal architecture revision) |
|----------|------------------------------------------------------|
| Backend integration, sync, admin data reads | Changes to scoring methodology |
| Provider selection and adapter implementations | Questionnaire content or structure |
| Identity and authorization policies | Report wording or Urdu copy |
| Canonical schema extensions (versioned) | UI visual redesign |

Any proposed change that violates a rule in this section must be documented as an architectural revision proposal. Implementation must not proceed until a new blueprint version is approved.

### 14.2 Domain Rules

The **Domain Layer owns all business behaviour**. Backend providers must never implement business logic.

Domain responsibilities include (non-exhaustive):

| Domain Service | Must remain backend-independent |
|----------------|--------------------------------|
| Assessment Engine | Yes |
| Scoring Engine | Yes |
| Report Generation | Yes |
| Recommendation Engine | Yes |
| Trend Engine | Yes |
| Community Insights | Yes |
| Average Implementation calculation | Yes |
| Participant lifecycle rules | Yes |
| Future analytical engines | Yes |

**Enforcement:**

- Domain modules must not import from `src/backend/providers/` or any provider SDK.
- Domain modules must not import Identity or Storage adapter implementations.
- Calculations performed in the Domain Layer are authoritative. No other layer may recompute them for display or persistence.

### 14.3 Backend Rules

The backend exists **only** to:

- Persist data
- Retrieve data
- Authenticate identities
- Authorize access

The backend must **never**:

- Calculate scores
- Produce reports
- Compute analytics (including community averages, trends, or recommendations)
- Generate recommendations
- Interpret questionnaire answers

**Enforcement:**

- Storage adapters return canonical entities or raw counts/timestamps. Domain Layer performs all derived calculations.
- Identity adapters establish sessions and validate roles. They do not evaluate assessment content.
- Provider security rules enforce access policy. They do not encode business formulas.

### 14.4 Presentation Rules

The Presentation Layer must **never**:

- Calculate implementation scores or percentages
- Apply business rules (scoring, averaging, trend detection, recommendation selection)
- Query provider-specific APIs directly (Firestore, Supabase client, REST endpoints)
- Know which backend provider is active
- Branch on `BACKEND_PROVIDER` or equivalent configuration

Presentation consumes **Application Layer outputs only** — rendered reports, formatted widget values, orchestrated navigation state.

**Enforcement:**

- `app.js` and `src/admin/*.js` (UI modules) import `application.js` or presentation helpers only.
- No presentation file imports from `src/backend/providers/`.
- Admin widget display values arrive pre-computed or pre-formatted from Application/Domain via `admin-dashboard-data.js`.

### 14.5 Canonical Model Rules

The **canonical data model is the single source of truth**. Provider schemas are mappings — not alternate definitions of meaning.

Providers must **never** redefine:

- Entity meaning (what a Participant or Assessment represents)
- Business relationships (e.g. one assessment belongs to one participant)
- Score semantics (`overallPercentage`, `implementationScore`)
- Report structure (assessment, categories, insights, recommendations)

**Enforcement:**

- All new persistable fields are added to `backend-schema.js` first, with documented semantics.
- Provider mapping modules translate field names and types. They do not add computed business fields at the provider layer.
- `schemaVersion` increments when canonical meaning changes. Providers migrate data; they do not reinterpret it.

### 14.6 Provider Independence Rules

Adding a new backend provider must require **only**:

1. Identity Adapter implementation
2. Storage Adapter implementation
3. Configuration registration in the provider registry

Adding a provider must **not** require changes to:

- Domain Layer
- Application Layer
- Presentation Layer (UI)
- Report generation
- Questionnaire content or flow
- Scoring engine

**Verification gate for new providers:** All Section 12 tests pass without modification to Domain, Application, or Presentation code.

### 14.7 Business Rule Protection

The following business rules are **locked**. Future contributors must preserve them unless a formal methodology revision is approved separately from this architecture.

| Rule | Definition |
|------|------------|
| **Latest assessment per participant** | One participant contributes exactly one score to community Average Implementation — the score from their most recent completed assessment. Historical assessments for the same participant are excluded. |
| **Score origin** | Overall implementation percentage originates **only** from the Scoring Engine (`overallPercentage = raw/max × 100`, one decimal). No other layer may derive or override this value. |
| **implementationScore identity** | `implementationScore` is identical to `overallPercentage`. It is not a separate weighted, normalized, or report-specific metric. |
| **Report immutability of calculations** | Reports display Domain calculations. Reports never recalculate scores, levels, or percentages at render time. |
| **Community metrics meaning** | Community metrics represent **current implementation levels** (latest state per participant for averages; anonymous bucket counts for aggregates) — not raw submission volume or historical totals across all assessments per participant. |
| **Canonical schema authority** | Business meaning is defined by the canonical model and Domain Layer. Provider storage formats are representations, not definitions. |
| **Anonymous community aggregates** | Community aggregate entities store bucket counts only. Individual participant answers are not stored in community-facing aggregate records. |
| **Assessment immutability after submit** | Completed assessments are not updated or deleted by participants. Administrators have read access but do not modify submitted assessment records. |

### 14.8 Future Extension Policy

New functionality must be added in the correct layer:

| Change Type | Correct Layer | Incorrect Layer |
|-------------|---------------|-----------------|
| New report section or insight | Domain (report/insights engines) | Storage adapter |
| New admin dashboard widget metric | Domain (analytics calculation) + Application (orchestration) + existing admin UI patterns | Provider SQL/query in UI |
| New persistable entity | Canonical schema → Storage adapter | Provider-only schema |
| New backend provider | Identity + Storage adapters + registry | Application or Domain |
| New authentication method | Identity Adapter | Storage adapter or UI |
| New analytical engine | Domain (new module) | Backend provider |
| New authorization role | Canonical security policy → provider enforcement | Hard-coded in UI |

**Extension workflow:**

1. Define business behaviour in Domain Layer.
2. Extend canonical schema if new data is persisted (increment `schemaVersion` if needed).
3. Extend Storage or Identity adapter **interfaces** if new operations are required.
4. Implement operation in each active provider (or stub for inactive providers).
5. Wire through Application Layer. Presentation receives outputs only.
6. Add tests per Section 12. Verify no governance rule in this section is violated.

### 14.9 Architecture Revision Process

To change any rule in Section 14:

1. Document the proposed change and rationale.
2. Assess impact on Domain, Application, Presentation, providers, and locked business rules.
3. Publish a new blueprint version (e.g. v2.3).
4. Obtain explicit approval before implementation.

Silent drift — implementing behaviour outside these rules without revision — is not permitted.

## Appendix: Optional Future Architecture (Event Layer)

> **Informational only.** This appendix describes a possible future evolution. It is **not** part of Version 2.2. Current implementation remains **synchronous**.

Today, assessment submission follows a synchronous path: Domain produces results → Application orchestrates → Storage Adapter persists. This is sufficient for Version 2.2 and the initial provider implementations.

A future version may introduce an **Event Layer** between Domain and infrastructure:

```text
Assessment Submitted
        │
        ▼
   Domain Event
        │
 ┌──────┼───────────────┐
 │      │               │
Storage Analytics Notifications
```

### Potential responsibilities

| Consumer | Purpose |
|----------|---------|
| **Storage** | Persist assessment and aggregates (current Storage Adapter role) |
| **Analytics** | Asynchronous aggregate recomputation, batch reporting |
| **Notifications** | Admin alerts, participant reminders, organizational digests |

### Constraints on future event architecture

If an Event Layer is introduced in a future version, the following must remain true:

- Domain Layer still owns all business calculations before events are emitted.
- Events carry canonical entities or domain value objects — not provider-native formats.
- Presentation and Application layers do not subscribe to provider SDKs directly.
- Section 14 governance rules remain in force.

Version 2.2 does not implement events, message queues, or async processing pipelines. Adapters persist synchronously (non-blocking from the participant's perspective, but without an event bus).
