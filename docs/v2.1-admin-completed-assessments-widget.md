# Version 2.1 — Admin Live Widget: Completed Assessments

Phase 2 connects **one** additional dashboard widget to Firestore.

## Widget

| Label | Source | Status |
|-------|--------|--------|
| Completed Assessments | `assessments/` document count | Live when `USE_FIREBASE: true` |

All other dashboard cards except **Total Participants** (Phase 1) remain sample data.

## Feature flag

| Flag | Behavior |
|------|----------|
| `USE_FIREBASE: false` | Sample `summary.completedAssessments` |
| `USE_FIREBASE: true` | Live count via `admin-firestore-service.js` |

When live mode is enabled, this widget never shows sample data.

## Architecture

```
admin-dashboard.js
        ↓
admin-dashboard-data.js
        ↓
admin-firestore-service.js → loadCompletedAssessmentsWidgetData()
        ↓
Firestore: assessments/
```

UI components never query Firestore directly.

## Data source

Collection: `assessments/`

Each document represents one submitted assessment.

| Counted | Not counted |
|---------|-------------|
| Assessment documents in `assessments/` | Participant documents |
| Multiple assessments per participant | Anonymous auth sessions |

## Firestore query

Primary (preferred):

```javascript
getCountFromServer(collection(db, "assessments"))
```

Fallback (documented temporary):

```javascript
getDocs(collection(db, "assessments")) → snapshot.size
```

Used only if `getCountFromServer` is unavailable in the runtime SDK import.

Implementation: shared `countCollectionWidgetData()` in `admin-firestore-service.js`.

## Loading behavior

1. Dashboard renders with `Loading...` for Completed Assessments
2. `loadDashboardLiveWidgets()` fetches Total Participants and Completed Assessments in parallel
3. Card updates with count + `Last updated: {timestamp}` on success

## Empty state

| Value | Meta |
|-------|------|
| `0` | `No assessments submitted yet.` |

## Error handling

| Status | Value | Meta |
|--------|-------|------|
| `permission-denied` | `—` | `Permission denied.` |
| `unavailable` | `—` | `Firestore unavailable.` |
| `error` | `—` | `Unable to load assessment count.` |

Errors are caught in the service layer. The dashboard never crashes.

## Security requirements

Admins require read access to count assessments:

```
allow read: if (own uid) || isGoogleAdmin();
```

Participant rules unchanged:

- Participants can still read only their own assessments
- Participants cannot read other users' assessments
- Create rules unchanged (immutable assessments)

Deploy updated rules:

```bash
firebase deploy --only firestore:rules
```

## Performance notes

- Prefer `getCountFromServer` — server-side aggregation, no document downloads
- Fallback scans all assessment documents — acceptable for early scale only
- Phase 1 and Phase 2 widget loads run in parallel via `Promise.all`

## Indexes

No composite index required for full-collection count aggregation.
