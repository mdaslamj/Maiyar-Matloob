## Root Cause

The Admin Gate treated `isBackendInfrastructureReady()` failure as “Firebase required” for every provider. When `BACKEND_PROVIDER` was `none` or `supabase`, users still saw **Admin Access Requires Firebase** and could not reach `/admin` in development mode.

The gate did not consult the backend provider abstraction — it assumed Firebase was mandatory for all admin access.

---

## Solution

Added `resolveAdminGateMode()` in `admin-auth-gate.js`, using `getBackendProviderName()` from `backend-service` and `FEATURE_FLAGS.USE_BACKEND_SYNC` (not Firebase flags).

| Mode | Condition | Behaviour |
|------|-----------|-----------|
| `development` | `USE_BACKEND_SYNC === false` or provider `none` | Direct `/admin` access; banner: **Development Mode — Authentication Disabled** |
| `firebase-required` | Provider `firebase`, backend not ready | Existing Firebase configuration screen (unchanged) |
| `firebase-auth` | Provider `firebase`, backend ready | Existing Google sign-in flow (unchanged) |
| `auth-not-configured` | Provider `supabase`, `rest`, or `self-hosted` | Neutral blocking message; no Firebase/Google references |

`admin-app.js` routes initialization through gate mode instead of a single Firebase readiness check.
